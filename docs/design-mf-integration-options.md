# MF連携の全体設計 — 現状分析と将来案

**作成日**: 2026-03-26
**目的**: 購買システム導入後のMFビジネスカード・MF経費・MF会計Plusの最適な連携モデルを検討

---

## 1. 現状の連携パターン

### パターン①: 管理本部のカード利用

```
管理本部がMFカードで決済
  → MFビジネスカード → MF会計Plus（直接API連携）
  → MF会計Plusの「通帳・カード他」に明細が出る
  → 管理本部がMF会計Plus上で仕訳作成
```

### パターン②: 従業員のカード利用

```
従業員がMFカードで決済
  → MFビジネスカード → MF経費（明細連携）
  → 従業員がMF経費で証憑添付 + 経費申請提出
  → 承認者がMF経費で承認
  → MF経費 → MF会計Plus（仕訳連携）
```

### 現状の問題点

- 従業員は「MFビジネスカードアプリ → MF経費で申請」の操作が必要
- 購買の承認はメール/口頭で行われており、MF経費の承認とは別
- 証憑管理がMF経費に依存（購買台帳との紐付けがない）

---

## 2. 購買システム導入後に生まれる経路

```
購買システム（新）:
  /purchase → Slack承認 → カード決済 → 検収 → 証憑添付
  → MF会計Plus API で仕訳作成（Sprint 3実装済み）

MFビジネスカード（既存）:
  カード決済 → MF会計Plusに明細が自動連携
  カード決済 → MF経費に明細が自動連携
```

**同じ1回の購入に対して、3つの経路からデータが流れる可能性がある:**

```
                    購買システム ──→ MF会計Plus（API仕訳）
                   /
MFカード決済 ────
                   \
                    MFビジネスカード ──→ MF会計Plus（明細自動連携）
                                   └─→ MF経費（明細自動連携）
```

---

## 3. 将来案の比較

### 案A: MF経費を維持（現行踏襲 + 購買システム追加）

```
購買の場合:
  購買システム → Slack承認 → 証憑管理
  MFカード → MF経費 → 従業員が経費申請 → MF会計Plus

購買以外の経費:
  MFカード → MF経費 → 従業員が経費申請 → MF会計Plus
```

| メリット | デメリット |
|---------|-----------|
| MF経費の既存運用を変えなくていい | 従業員が購買でもMF経費で申請が必要（二重作業） |
| MF経費の承認フローがそのまま使える | 購買システムの承認とMF経費の承認が二重 |
| | 仕訳が二重に作られるリスク |

---

### 案B: 購買はMF経費バイパス、経費のみMF経費

```
購買の場合:
  購買システム → Slack承認 → 証憑管理 → MF会計Plus API（直接仕訳）
  MFカード → MF会計Plus（明細自動連携）→ 対象外にする or 照合
  MFカード → MF経費 → ★ 購買分は申請不要（放置 or 連携停止）

購買以外の経費（出張・接待・日常）:
  MFカード → MF経費 → 従業員が経費申請 → MF会計Plus
```

| メリット | デメリット |
|---------|-----------|
| 購買は証憑添付1回で完了 | MF経費に購買分の明細が残る（放置 or 対象外処理が必要） |
| 承認は購買システム1箇所 | MF会計Plusのカード明細も対象外処理が必要 |
| 仕訳の二重作成がない | MF経費での「購買」と「経費」の仕分けが必要 |

---

### 案C: MF経費を経費全般に統一（購買含む）

```
全カード利用:
  購買システム → Slack承認 → 証憑管理（OCR + 突合）
  MFカード → MF経費 → ★ 証憑は購買システムから自動転送
                     → 経費申請は管理本部が一括作成/承認（API）
                     → MF会計Plus

購買以外の経費:
  MFカード → MF経費 → 従業員が経費申請 → MF会計Plus
```

| メリット | デメリット |
|---------|-----------|
| MF経費を全カード利用の一元管理に使える | MF経費APIに申請作成がない（調査済み）|
| 会計監査でMF経費が証跡になる | 従業員のMF経費操作は不要にできるが、管理本部の作業が増える |
| | API制約で完全自動化は不可能 |

---

### 案D: MF会計Plus一本化（MF経費は段階的に縮小）

```
全カード利用:
  購買システム → Slack承認 → 証憑管理 → MF会計Plus API（直接仕訳）
  MFカード → MF会計Plus（明細自動連携）→ 購買システムの仕訳と照合

  ★ MF経費は使わない（購買も経費も全て購買システム経由）

出張・日常経費:
  /trip → Slack → 管理本部が MF会計Plus に直接仕訳
  日常経費 → /expense コマンド（将来実装）→ MF会計Plus直接
```

| メリット | デメリット |
|---------|-----------|
| システムが最もシンプル | MF経費の契約/ライセンスが無駄になる |
| 従業員の操作が1箇所に集約 | 出張・接待等のフォーム/ワークフローを自作する必要あり |
| MF経費の申請/承認の二重作業がゼロ | MF経費のOCR・交通費計算等の便利機能が使えなくなる |
| 全仕訳が購買システム経由で追跡可能 | 実装範囲が大きい |

## 5. 推奨: 案B → 案Dへの段階的移行

### Phase 1（今回）: 案B — 購買はMF経費バイパス

```
購買: 購買システム → MF会計Plus直接
経費: MF経費 → MF会計Plus（現行維持）
```

- 購買の従業員操作を1回に
- MF経費の既存運用は変えない
- MF会計PlusとMF経費の両方にカード明細が来る問題は運用ルールで対処

**管理本部の対応:**
- MF経費に来る購買分の明細 → 無視 or 「購買システム処理済み」タグ
- MF会計Plusに来るカード明細 → 購買システムの仕訳と照合し「対象外」

### Phase 2（将来）: 案D方向 — 経費も購買システムに統合

```
/expense コマンドを追加（出張・接待・日常経費）
→ 全てのカード利用を購買システムで管理
→ MF経費を段階的に縮小
```
