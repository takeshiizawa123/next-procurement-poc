# カード明細マッチング設計書

**作成日**: 2026-03-27（改訂）
**目的**: 購買システムの仕訳（Stage 1）とMFビジネスカードの利用明細を照合し、PO番号を紐付ける

---

## 1. 基本方針

### カード明細 = Stage 2仕訳として活用する

```
MF会計Plus上の自動仕訳ルール:
  カード明細が取り込まれた時:
    借方: 未払金(MFカード:未請求)
    貸方: 未払金(MFカード:請求)
    → これがStage 2の振替仕訳そのもの

購買システム:
  GET /journals?entered_by=none で Stage 2仕訳を取得
  → Stage 1（購買台帳）とマッチング
  → マッチしたらPO番号を紐付け
  → マッチしなかったもの = 一般経費 or 要調査
```

### なぜこの方式か

```
従来の問題:
  Stage 1（購買システム） + カード明細 → 二重計上リスク
  → カード明細を「対象外」にする運用が必要 → 手間

新しい方式:
  カード明細を「対象外」にせず、Stage 2振替仕訳として登録
  → 二重計上にならない（未請求→請求の振替なので）
  → CSVの手動エクスポートも不要
  → 購買システムはマッチングとPO番号紐付けだけ担当
```

### 仕訳の全体像

```
                    entered_by    仕訳
                    ──────────    ────
Stage 1（証憑添付時）:  api        費用 / 未払金(未請求)    ← 購買システムがPOST
Stage 2（明細到着時）:  none       未払金(未請求) / 未払金(請求) ← 自動仕訳ルールで登録
Stage 3（引落時）:     none       未払金(請求) / 普通預金    ← 自動仕訳ルールで登録
```

## 2. データソースと取得方法

### 2.1 購買側データ（Stage 1）

**取得元**: GASスプレッドシート（購買台帳）
**取得方法**: GAS API

```
購買台帳から取得するフィールド:
  po_number:       PO-2026-0042
  amount:          52,800（証憑の金額 = Stage 1 仕訳額）
  transaction_date: 2026-03-15（証憑の取引日）
  supplier_name:   Amazon
  payment_method:  会社カード
  stage1_journal_id: 12345（MF会計Plusの仕訳ID）
  stage:           1（Stage 1 完了済み）
  card_matched:    false（未照合）
```

### 2.2 カード明細データ（Stage 2仕訳）

**取得元**: MF会計Plus API
**取得方法**: `GET /journals?entered_by=none&start_date=...&end_date=...`
**前提**: MF会計Plusの自動仕訳ルールで、カード明細を以下の仕訳で自動登録する設定が必要

```
自動仕訳ルール設定:
  借方: 未払金(MFカード:未請求)
  貸方: 未払金(MFカード:請求)
  摘要: カード明細の加盟店名（自動反映）
```

**APIレスポンスから取得できるフィールド**:

```
GET /journals?entered_by=none で取得される各仕訳:
  id:               99001（MF会計PlusのStage 2仕訳ID）
  transaction_date:  2026-03-15（カード利用日）
  branches[0]:
    remark:          "AMAZON.CO.JP"（加盟店名 = 摘要）
    debitor.value:   52800（カード利用額）
    creditor.value:  52800
  approval_status:   "draft" or "approved"
```

### 2.3 データ取得の整理

| データ | 取得元 | 方法 | 手動作業 |
|--------|--------|------|---------|
| Stage 1（購買） | GAS購買台帳 | GAS API | なし |
| Stage 2（カード明細） | MF会計Plus | `GET /journals?entered_by=none` | **なし** |

**CSVエクスポートは不要。全自動。**

## 3. マッチングアーキテクチャ

### 3.1 全体フロー

```
月次バッチ or 手動実行（カード確定後）

購買システム（自動）:
  ① GET /journals?entered_by=none → Stage 2仕訳リストを取得
  ② GAS購買台帳から「会社カード + Stage 1済み + 未照合」を取得
  ③ マッチングアルゴリズム実行
  ④ 結果を3分類:
     ├── 確信マッチ → GAS更新（PO番号紐付け + Stage=2）
     ├── 候補マッチ → 経理に確認依頼
     └── 未マッチ（カード側） → 一般経費 or 要調査
     └── 未マッチ（購買側） → 翌月繰越 or キャンセル確認

経理:
  ⑤ マッチング結果を確認（候補マッチの承認/修正）
  ⑥ MF会計Plusで Stage 2仕訳を承認（必要に応じて）
```

### 3.2 マッチングの入力

```
左側: 購買レコード（GAS）
  [{
    po: "PO-2026-0042",
    amount: 52800,
    date: "2026-03-15",
    supplier: "Amazon",
    stage1JournalId: 12345
  }, ...]

右側: Stage 2仕訳（MF会計Plus API）
  [{
    journalId: 99001,
    date: "2026-03-15",
    amount: 52800,
    remark: "AMAZON.CO.JP"
  }, ...]
```

## 4. マッチングアルゴリズム

### 4.1 スコアリング方式

各購買レコード × 各カード明細の組み合わせにスコアを付与し、最適な1:1マッチングを決定する。

```
totalScore = amountScore + dateScore + nameScore
```

#### 金額スコア（最重要・最大50点）

```
差額 = |購買金額 - カード金額|
差額率 = 差額 / 購買金額

金額一致（差額 = 0）:           50点
差額率 ≤ 0.01（1%以内）:        40点  ← 税端数・為替微差
差額率 ≤ 0.05（5%以内）:        25点  ← 送料追加等
差額率 ≤ 0.10（10%以内）:       10点  ← 要確認
差額率 > 0.10:                   0点  ← マッチしない
```

#### 日付スコア（最大30点）

```
日差 = |購買日 - カード利用日|（日数）

同日:              30点
1日差:             25点
2日差:             20点
3日差:             15点
4-7日差:           5点   ← 海外EC等で遅延あり得る
8日以上:           0点   ← マッチしない
```

#### 取引先名スコア（最大20点）

```
購買側: "Amazon"
カード側: "AMAZON.CO.JP"

完全一致:                           20点
正規化後一致:                       18点  ← 大文字小文字・.CO.JP除去
どちらかがもう一方を含む（部分一致）: 15点  ← "Amazon" ⊂ "AMAZON.CO.JP"
先頭3文字一致:                      8点   ← "AMA" = "AMA"
一致なし:                           0点
```

### 4.2 取引先名の正規化

```typescript
function normalizeMerchantName(name: string): string {
  return name
    .toUpperCase()                      // 大文字統一
    .replace(/\s+/g, "")               // 空白除去
    .replace(/\.CO\.JP$/i, "")         // .CO.JP除去
    .replace(/\.COM$/i, "")            // .COM除去
    .replace(/（.*?）/g, "")           // 全角括弧内除去
    .replace(/\(.*?\)/g, "")           // 半角括弧内除去
    .replace(/[株式会社|㈱|（株）]/g, "")  // 法人格除去
    .replace(/[ー－\-]/g, "");         // ハイフン類除去
}
```

### 4.3 よくある加盟店名の対応表（設定可能）

```
加盟店名エイリアス辞書（管理画面で編集可能）:

| 購買台帳の取引先名 | カード明細に出る加盟店名パターン |
|------------------|-------------------------------|
| Amazon           | AMAZON.CO.JP, AMAZON DOWNLOADS, AMAZONマーケットプレイス |
| モノタロウ         | MONOTARO, モノタロウ, MONO TARO |
| ヨドバシカメラ      | ヨドバシ.COM, YODOBASHI, ヨドバシカメラ |
| Askul            | ASKUL, アスクル, ASKUL.CO.JP |
| Rakuten          | 楽天市場, RAKUTEN, 楽天ペイ |
```

### 4.4 マッチング判定

```
確信マッチ:  totalScore ≥ 80  → Stage 2 仕訳を自動生成
候補マッチ:  totalScore 50-79 → 経理に確認依頼
未マッチ:    totalScore < 50  → 照合不能アラート
```

### 4.5 1:1マッチングの最適化

同じカード明細に複数の購買がマッチする場合（同日同額の複数購入等）:

```
手順:
1. 全ペアのスコアを計算
2. スコア降順でソート
3. 上位から順に採用（既に使用済みのレコードはスキップ）
4. 同スコアの場合は日付が近い方を優先

例:
  購買A: 3/15 ¥5,500 Amazon
  購買B: 3/15 ¥5,500 Amazon
  カード①: 3/15 ¥5,500 AMAZON.CO.JP
  カード②: 3/16 ¥5,500 AMAZON.CO.JP

  A × ① = 100点（日付一致+金額一致+名前一致）
  A × ② = 95点（日付1日差+金額一致+名前一致）
  B × ① = 100点
  B × ② = 95点

  → A-①を採用（先に出現）→ B-②を採用
```

## 5. マッチング結果の処理

### 5.1 確信マッチ（金額一致）

```
購買: PO-2026-0042, ¥52,800（Stage 1 金額）
Stage 2仕訳（自動登録済み）: ¥52,800
差額: ¥0

処理:
  → GAS購買台帳を更新:
    stage: 2
    card_matched: true
    stage2_journal_id: 99001
  → 追加仕訳は不要（Stage 2仕訳は自動仕訳ルールで既に登録済み）
  → マッチング結果をログに記録
```

### 5.2 確信マッチ（差額あり）

```
購買: PO-2026-0043, ¥33,000（Stage 1 金額）
Stage 2仕訳（自動登録済み）: ¥33,550
差額: +¥550

処理:
  → Stage 2仕訳は ¥33,550 で既に登録されている（正しい）
  → Stage 1の費用計上額と差額があるので、差額調整仕訳を追加作成:

  POST /journals:
    tags: ["PO-2026-0043", "Stage2-adj"]
    memo: "PO-2026-0043 差額調整（カード確定値との差 +¥550）"
    branches:
      - remark: "PO-2026-0043 モノタロウ 差額調整（送料追加）"
        debitor:
          account_code: "消耗品費"    ← Stage 1と同じ費用科目
          value: 550
        creditor:
          account_code: "未払金"
          sub_account_code: "MFカード:未請求"
          value: 550

  → これによりStage 1の未請求残高が ¥33,550 に増え、
    Stage 2の振替額 ¥33,550 と一致する
```

### 5.3 一般経費の自動仕訳ルール問題（★重要）

```
問題:
  自動仕訳ルールは全カード明細に適用される
  → 購買分: 未払金(未請求)/未払金(請求) = 正しい（Stage 2振替）
  → 一般経費: 未払金(未請求)/未払金(請求) = 間違い（費用計上されない）

解決策（推奨）: 購買でよく使う加盟店のみ振替ルールを設定
  → "AMAZON"を含む → 未払金(未請求)/未払金(請求)
  → "MONOTARO"を含む → 同上
  → "ASKUL"を含む → 同上
  → デフォルト（上記以外）→ 仕訳候補のまま（経理が従来通り処理）

結果:
  購買でよく使う加盟店 → 自動でStage 2仕訳登録 → APIで取得可能
  一般経費（タクシー等）→ 仕訳候補のまま → 経理がUIで処理
```

### 5.4 未マッチのStage 2仕訳

```
自動仕訳ルールで登録されたが、購買台帳にマッチしないStage 2仕訳

パターン1: 同じ加盟店で購買以外の利用
  → 例: 個人的なAmazon購入を会社カードで決済
  → 経理がMF会計Plusで借方を費用科目に修正（未払金:未請求 → 消耗品費等）

パターン2: 購買だが加盟店名がマッチしなかった
  → アラート → 経理が手動照合
  → 加盟店名エイリアス辞書に追加

パターン3: 不明な利用
  → 要調査
```

### 5.5 未マッチの購買レコード

```
購買台帳にあるがStage 2仕訳にマッチしない案件

原因:
  → カード確定値がまだ出ていない（翌月繰越）
  → 加盟店が自動仕訳ルールに未登録（仕訳候補のまま残っている）
  → キャンセル・返品済み

処理:
  → 翌月に再照合
  → 2ヶ月連続未マッチはアラート
  → 加盟店ルール未登録の場合は、経理にMF会計Plusの仕訳候補を確認依頼
```

## 7. MF会計Plusのカード明細仕訳候補の処理

### 7.1 方針

**マッチング済みのカード明細 → MF会計Plusで「対象外」にする**

理由: Stage 2仕訳を購買システムがAPI経由で作成するため、カード明細からの仕訳候補は不要。二重計上を防止。

### 7.2 「対象外」処理の効率化

```
方法A: 自動仕訳ルールで事前に対策
  → MF会計Plusの自動仕訳ルールで、よく使う加盟店を
    「登録しない」（対象外）に設定
  → 設定例:
    - "AMAZON" を含む → 対象外
    - "MONOTARO" を含む → 対象外
    - "ASKUL" を含む → 対象外
  → 問題: 購買以外のAmazon利用（個人の一般経費）も対象外になる

方法B: 手動で月次対象外処理
  → マッチング結果の確信マッチリストを見ながら
    MF会計Plusで1件ずつ「対象外」をクリック
  → 月30件なら5-10分

方法C（推奨）: ハイブリッド
  → 購買専用のカード（特定の番号下4桁）がある場合:
    そのカード番号の明細を一括「対象外」にするルール
  → カードが共用の場合:
    マッチング結果を元に手動で対象外処理（方法B）
```

## 10. エッジケースと対処

| # | ケース | 対処 |

| 1 | 同日同額同店の複数購入 | カード名義人（購入者）で追加判別。不可能なら候補マッチに回す |
| 2 | 分割払い | カード明細の金額が月割りになる。購買台帳の金額と一致しない → 候補マッチ or 手動 |
| 3 | 海外EC（為替差） | 差額率5%以内なら確信マッチ。それ以上は候補マッチ |
| 4 | 返品（マイナス明細） | カード明細にマイナス行 → 購買台帳のキャンセル案件と照合 |
| 5 | カード利用の翌月確定 | 締日をまたぐ利用は翌月CSVに出現。未マッチ購買は翌月に自動繰越 |
| 6 | ポイント利用による減額 | 差額が小さければ確信マッチ。差額調整仕訳に「ポイント値引」と記載 |
| 7 | 1回の決済に複数PO | PO分割されていない → 購買台帳の合計額でマッチ or 手動紐付け |
| 8 | 個人利用の混在 | 未マッチカード明細に残る → 経理が判断 |
