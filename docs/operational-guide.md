## 1. 登場人物と日常業務

### 3つの役割

| 役割 | 例 | 主な操作 | MFカード |
|------|---|---------|---------|
| 申請者 | 営業部 田中 | Slack `/purchase` で申請、カード決済、検収、証憑添付 | バーチャルカード保有 |
| 承認者（部門長） | 営業部 山田部長 | Slack DM で承認/差戻し（ボタン1クリック） | バーチャルカード保有 |
| 管理本部 | 鈴木 | Slack + Webで発注・仕訳・支払・統制 | あり |

### 1日の流れ

**申請者の1日:**
- 業務中: `/purchase` → モーダル入力（2-3分）→ 承認待ち
- 承認後: MFカードで購入（10万未満は自分で発注）
- 物品到着: [検収完了] ボタン（10秒）→ スレッドに証憑PDF添付（30秒）

**承認者の1日:**
- 随時: Bot DMの [承認] / [差戻し] ボタンを押す（10秒/件）
- 承認時に申請者の未提出証憑一覧が表示される

**管理本部の1日:**
- 09:00: #purchase-ops の日次サマリを確認
  - 🔴要対応（承認待ち・発注待ち）
  - 🟡フォロー要（3日以上停滞の証憑待ち）
  - 🟢順調（進行中・完了件数）
- 午前: 「発注待ち」案件の処理（高額・請求書払いのみ管理本部が発注）
- 午後: 「仕訳待ち」案件のMF会計Plus登録
- 週次: カード明細突合レポートの確認

## 2. 購買パターン（4パターン）

### パターンA: 少額カード購入（10万円未満）— 大多数

```
申請者: /purchase → モーダル入力
  ↓
Bot → #purchase-request に投稿 + 部門長DMに承認依頼
  ↓
部門長: [承認] ボタン
  ↓
Bot → 申請者DM: 「承認されました。発注してください」
  ↓ ★申請者が自分でカード決済
申請者: Amazon等で購入 → [発注完了] ボタン
  ↓
物品到着 → 申請者: [検収完了] ボタン
  ↓
Bot: 「証憑をスレッドに添付してください」
  ↓
申請者: 納品書PDFをドラッグ&ドロップ
  ↓
Bot: OCR自動解析 → 「金額一致 / 種別: 納品書」確認
  ↓
管理本部: MF会計Plusに仕訳登録 → 完了
```

**所要期間**: 3-4日 / **承認**: 1回（部門長のみ）

---

### パターンB: 高額購入（10万円以上）— 管理本部が発注

```
申請者: /purchase → ¥350,000 サーバー機器
  ↓
部門長: [承認] → Bot → 管理本部DM: 「二段階承認依頼」
  ↓
管理本部: [承認] → Bot → #purchase-ops: 「発注待ち」
  ↓ ★管理本部がカード決済
管理本部: 発注 → [発注完了]
  ↓
以降はパターンAと同じ（検収→証憑→仕訳）
```

**承認**: 2回（部門長 + 管理本部）/ **発注者**: 管理本部

---

### パターンC: 請求書払い — 管理本部が発注

```
申請者: /purchase → 支払方法: 「請求書払い」
  ↓
承認フロー（金額に応じて1段階 or 2段階）
  ↓
Bot → #purchase-ops: 「発注待ち（請求書払い）」
  ↓ ★管理本部が発注書を送付
管理本部: [発注完了]
  ↓
請求書到着 → 証憑として添付 → 仕訳 → 支払
```

---

### パターンD: 購入済（立替）— 緊急時のみ

```
申請者: /purchase → 申請区分: 「購入済」+ 証憑添付
  ↓
部門長: [承認]
  ↓ ★承認・発注・検収をスキップ → 即「証憑待ち」
Bot: 証憑確認 → 管理本部が仕訳 → 立替金精算
```

**条件**: 証憑添付が必須 / 事前にカード決済すべきだった場合の救済措置

## 3. 承認ルールと閾値

| 金額 | 承認段階 | 発注者 | 根拠 |
|------|---------|--------|------|
| 10万円未満 | 部門長のみ | 申請者（カード）/ 管理本部（請求書） | — |
| 10万円以上 | 部門長 → 管理本部 | 管理本部 | 固定資産計上基準 |

**権限チェック（厳格）:**
- [承認] ボタン → 部門長のみ（従業員マスタの `deptHeadSlackId` で検証）
- [発注完了] → 10万以上/請求書払いは管理本部のみ（fail-closed）
- [検収完了] → 申請者のみ
- [取消] → 申請者のみ（発注前に限る）

## 4. 証憑管理（3層アーキテクチャ）

### 第1層: システムブロック

**原則: 証憑なし = 会計処理に進めない**

- 仕訳登録API (`/api/mf/journal`) が証憑ステータスを確認
- 「添付済」でなければ仕訳登録を拒否

### 第2層: 段階催促（自動）

| 経過日数 | 時刻 | 方法 | 対象 |
|---------|------|------|------|
| Day 0 | 即時 | スレッド投稿 | 申請者 |
| Day 1 | 10:00 | DMダイジェスト（未提出まとめて1通） | 申請者 |
| Day 3 | 10:00 | スレッドに公開投稿（@メンション） | 全員に可視 |
| Day 7 | 10:00 | 部門長DMエスカレーション | 部門長 |

### 第3層: 抑止統制

- 承認時に申請者の未提出一覧を表示（部門長の目）
- Day 3の公開投稿（同僚にも見える）
- 月次で管理本部がカード突合結果をレビュー

## 5. 統制設計（3層統制）

### 事前統制（Preventive）

- 二段階承認（10万以上）
- 発注権限の制御（管理本部のみが高額/請求書案件を発注）
- 購入済申請は証憑必須

### 発見統制（Detective）

**カード明細突合（週次・月曜 11:00 自動実行）:**
- MF経費APIからカード明細取得 → 購買台帳と照合
- 検知パターン:
  - 未申請購入（明細あり + 申請なし）→ 🔴HIGH アラート
  - 承認前購入（明細日 < 承認日）→ 🟡MEDIUM
  - 金額不一致（明細 ≠ 申請額）→ 差額に応じてHIGH/MEDIUM

**OCR金額照合（証憑添付時に自動実行）:**
- Gemini Vision APIで証憑から金額抽出
- 申請金額と照合（許容差: ±¥500）
- 不一致 → 管理本部にアラート

**日次サマリ（毎朝 09:00 自動投稿）:**
- 要対応・フォロー要・順調の3区分
- 遅延案件はスレッドリンク付き

### 抑止統制（Behavioral Change）

- 段階催促による行動変容
- カード突合結果の部門長共有
- 月次コンプライアンスレビュー

## 7. 出張管理（/trip）

### 宿泊

**じゃらんJCS（先行導入）:**
- 従業員がじゃらんで予約（法人専用項目にプロジェクト番号を入力）
- 月次でCSVダウンロード → `/api/trip/import-csv` でMF経費に一括登録
- 請求書は郵送（紙） → スキャンしてMF会計Plusに添付

**楽天Racco（後追い導入）:**
- 与信審査完了後にじゃらんと併用 or 切替
- PDF請求書対応（スキャン不要）

### 航空券・新幹線

- ANA Biz / JAL Online で法人予約
- MFビジネスカードで決済 → MF経費に自動連携
- `/trip` コマンドで事前承認（Slackモーダル）

### /trip コマンドフロー

```
申請者: /trip → モーダル（行き先・日程・目的・交通手段・概算額）
  ↓
Bot → #出張チャンネルに投稿
  ↓
Bot → 申請者DM: 確認通知 + MF経費ID
  ↓
申請者: じゃらん/ANA/JALで予約
  ↓
月次: CSVインポート → MF経費 → MF会計Plus仕訳
```

## 9. ステータス遷移図

### 通常フロー（カード10万未満）

```
申請済 → 承認済 → 発注済 → 検収済 → 証憑完了 → 計上済 → 支払済
```

### 高額フロー（カード10万以上）

```
申請済 → 部門長承認済 → 管理本部承認済 → 発注済 → 検収済 → 証憑完了 → 計上済 → 支払済
```

### 購入済フロー（立替）

```
申請済 → 承認済 → 検収済（即時） → 証憑完了 → 計上済 → 支払済
```

### 差戻し・取消

```
申請済 → 差戻し（終了 — 新PO番号で再申請）
申請済 → 取消（終了）
承認済 → 取消（発注前に限る）
```
