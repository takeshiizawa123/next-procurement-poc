## 1. 登場人物と日常業務

### 3つの役割

| 役割 | 例 | 主な操作 | MFカード |
|------|---|---------|---------|
| 申請者 | 営業部 田中 | Slack `/purchase` で申請、カード決済、検収、証憑添付 | バーチャルカード保有 |
| 承認者（部門長） | 営業部 山田部長 | Slack DM で承認/差戻し（ボタン1クリック） | バーチャルカード保有 |
| 管理本部 | 鈴木 | Slack + Webで仕訳・照合・支払 | あり |

### 1日の流れ

**申請者の1日:**
- 業務中: `/purchase` → モーダル入力（2-3分）→ 承認待ち
- 承認後: MFカードで購入、または発注して請求書を提出
- 物品到着: [検収完了] ボタン（10秒）→ スレッドに証憑PDF添付（30秒）

**承認者の1日:**
- 随時: Bot DMの [承認] / [差戻し] ボタンを押す（10秒/件）
- 承認時に申請者の未提出証憑一覧が表示される

**管理本部の1日:**
- 09:00: #purchase-ops の日次サマリを確認
  - 🔴要対応（承認待ち・発注待ち）
  - 🟡フォロー要（3日以上停滞の証憑待ち）
  - 🟢順調（進行中・完了件数）
- 午前: 証憑・請求書の確認、仕訳待ち案件の処理
- 午後: 「仕訳待ち」案件のMF会計Plus登録
- 週次: カード明細突合レポートの確認

## 2. 購買パターン（3パターン）

### パターンA: カード払い — 大多数

```
申請者: /purchase → モーダル or Webフォームで入力
  ↓
Bot → #purchase-request に投稿 + 部門長DMに承認依頼
  ↓
部門長: [承認] ボタン
  ↓
Bot → 申請者DM: 「承認されました。カードで発注してください」
  ↓ ★申請者が自分でカード決済（金額問わず）
申請者: MFビジネスカード（VISA）で購入 → [発注完了] ボタン
  ↓
物品到着 → 申請者: [検収完了] ボタン
  ↓
Bot: 「証憑をスレッドに添付してください」
  ↓
申請者: 納品書PDFをSlackスレッドにD&D or マイページからアップロード
  ↓
Bot: OCR自動解析 → 「金額一致 / 種別: 納品書」確認
  ↓
管理本部: MF会計Plusに仕訳登録 → 完了
```

**所要期間**: 3-4日 / **承認**: 1回（部門長のみ）/ **発注**: 申請者

---

### パターンB: 請求書払い — VISAが使えない場合等

```
申請者: /purchase → 支払方法: 「請求書払い」
  ↓
部門長: [承認]
  ↓
Bot → 申請者DM: 「承認されました。発注してください。届いた請求書は管理本部に提出してください」
  ↓ ★申請者が自分で発注（電話・メール・Webサイト等）
申請者: [発注完了] ボタン
  ↓
物品到着 → 申請者: [検収完了] ボタン
  ↓
申請者: 届いた請求書をSlackスレッド or マイページから提出
  ↓
管理本部: 請求書確認 → 支払処理（銀行振込等）→ 仕訳登録
```

**承認**: 1回（部門長のみ）/ **発注**: 申請者 / **証憑**: 請求書

---

### パターンC: 購入済（立替精算）— 緊急時のみ

```
申請者: /purchase → 申請区分: 「購入済」+ 証憑添付
  ↓
部門長: [承認]
  ↓ ★発注・検収をスキップ → 即「証憑完了」
Bot: 証憑確認 → MF経費に証憑アップロード（自動）
  ↓
管理本部: MF経費で経費精算を確定 → 給与と合算して立替分を振込
```

**条件**: 証憑添付が必須 / MF経費経由で精算
**注意**: MF経費のカード明細から経費登録しないこと（二重計上防止）

## 3. 承認ルール

| 項目 | ルール |
|------|--------|
| 承認 | 部門長のみ（金額に関わらず1段階） |
| 発注 | 全件申請者が発注（カード・請求書問わず） |
| 高額品（10万以上） | 購入品の用途・理由の入力が必須 |

**権限チェック（厳格）:**
- [承認] ボタン → 部門長のみ（従業員マスタの `deptHeadSlackId` で検証）
- [発注完了] → 申請者・承認者・管理本部メンバー
- [✅ 全数検収] → 申請者または指定検収者
- [📦 部分検収] → 申請者または指定検収者
- [↩️ 返品] → 申請者・指定検収者・管理本部メンバー
- [取消] → 申請者のみ（発注前に限る）

## 4. 証憑管理（3層アーキテクチャ）

### 第1層: システムブロック

**原則: 証憑なし = 会計処理に進めない**

- 仕訳登録API (`/api/mf/journal`) が証憑ステータスを確認
- 「添付済」でなければ仕訳登録を拒否

### 第2層: 段階催促（自動）

| 経過日数 | 時刻 | 方法 | 対象 |
|---------|------|------|------|
| Day 0 | 即時 | スレッド投稿 | 申請者 |
| Day 1 | 10:00 | DMダイジェスト（未提出まとめて1通） | 申請者 |
| Day 3 | 10:00 | スレッドに公開投稿（@メンション） | 全員に可視 |
| Day 7 | 10:00 | 部門長DMエスカレーション | 部門長 |

### 第3層: 抑止統制

- 承認時に申請者の未提出一覧を表示（部門長の目）
- Day 3の公開投稿（同僚にも見える）
- 月次で管理本部がカード突合結果をレビュー

## 5. 統制設計（3層統制）

### 事前統制（Preventive）

- 部門長承認
- 高額品（10万以上）は用途・理由の入力が必須
- 購入済申請は証憑必須

### 発見統制（Detective）

**カード明細突合（週次・月曜 11:00 自動実行）:**
- MF経費APIからカード明細取得 → 購買台帳と照合
- 検知パターン:
  - 未申請購入（明細あり + 申請なし）→ 🔴HIGH アラート
  - 承認前購入（明細日 < 承認日）→ 🟡MEDIUM
  - 金額不一致（明細 ≠ 申請額）→ 差額に応じてHIGH/MEDIUM

**OCR金額照合・適格請求書検証（証憑添付時に自動実行）:**
- Gemini Vision APIで証憑から金額・税率・登録番号を抽出
- 申請金額と照合（許容差: ±¥500）→ 不一致 → 管理本部にアラート
- 税率（10%/8%軽減）を仕訳に反映
- 登録番号（T+13桁）を国税庁APIで検証 → 無効/未登録 → 管理本部にアラート

**固定資産通知（検収完了時に自動実行）:**
- 10万円以上の購買が検収完了 → OPSチャネルに「固定資産登録が必要です」通知
- 資産名・取得価額・取得日・部門・購入先を自動記載
- 管理本部はMF固定資産に手動登録

**日次サマリ（毎朝 09:00 自動投稿）:**
- 要対応・フォロー要・順調の3区分
- 遅延案件はスレッドリンク付き

**日次金額乖離アラート（毎日 12:00 自動実行）:**
- 予測明細（predicted_card_transactions）とカード実利用額を比較
- 検知パターン:
  - 金額乖離（差額 ±10% or ±¥1,000超）→ 🔴 OPSチャネルに通知
  - 未マッチ停滞（pending のまま3日以上）→ 🟡 OPSチャネルに通知
- エンドポイント: `POST /api/cron/daily-variance`

**従業員別利用傾向ダッシュボード（/admin/spending）:**
- 従業員別の購買金額・件数をランキング表示
- 部門別構成比のスタックバー
- 従業員ごとの月別推移・勘定科目別・支払方法別の内訳
- 期間選択: 1/3/6/12ヶ月
- 異常傾向（急増・偏り）の早期発見を目的とする

### 抑止統制（Behavioral Change）

- 段階催促による行動変容
- カード突合結果の部門長共有
- 従業員別利用傾向ダッシュボードによる可視化
- 月次コンプライアンスレビュー

### セキュリティ対策

- **Web UIログイン**: Google OAuth (NextAuth v5) / 内部ユーザータイプで `@futurestandard.co.jp` ドメインに制限
- **セッション管理**: NextAuth JWTセッション / 未認証時は `/auth/signin` にリダイレクト
- **API認証（管理系）**: Bearer token (`CRON_SECRET`) 認証
- **API認証（ブラウザ向け）**: `INTERNAL_API_KEY` 認証
- **Slack署名検証**: HMAC-SHA256 + タイミングセーフ比較 + 5分リプレイ防止
- **OAuth CSRF防止**: MF会計Plus OAuthコールバックでstateパラメータをcookieと照合
- **外部APIタイムアウト**: 全外部API呼出しに10-30秒のタイムアウト設定
- **DBリトライ**: Supabase接続失敗時はDrizzleが自動リトライ（PgBouncer経由）
- **Slack障害耐性**: 購買申請時にSlack投稿が失敗してもDB登録済みデータは保全
- **承認順序逆転対応**: Slackメッセージ更新前にDB更新を実行、失敗時はephemeralエラー（データ整合性）
- **ファイル制限**: 証憑アップロード上限10MB、対応形式: PDF/JPEG/PNG/HEIC/WebP/TIFF
- **トークン保護**: リフレッシュトークンは `mf_oauth_tokens` テーブルに保存、ログ出力なし
- **認証バイパス防止**: 環境変数未設定時はAPI認証をスキップせず500エラーを返す（localhostのみ例外）
- **承認権限チェック強化**: 承認者未設定時でもバイパスしない（管理者のみ承認可）
- **Slack署名必須化**: SIGNING_SECRET未設定時は全リクエストを拒否（スキップしない）
- **入力サニタイズ**: HTMLタグ除去、SQLインジェクションはDrizzle ORMで自動防止、500文字制限
- **数値バリデーション**: 金額1〜1億円、数量1〜10,000の範囲チェック
- **Slackイベント冪等性**: `slack_event_log` テーブルで重複処理防止（GAS時代は不可能だった）
- **仕訳冪等性**: PO番号による重複チェックで二重仕訳を防止
- **OAuth失敗アラート**: MF会計Plusトークン更新失敗時にOPSチャネルへ自動通知
- **カード情報警告**: カード下4桁が未登録の従業員の承認時にOPSへ警告

## 8. カード明細照合（月次）

### 8.1 概要

カード明細照合は、MFビジネスカードの利用明細と購買申請を突合し、未申請利用や申請漏れを検出する月次業務です。

```
照合UI: https://{vercel-url}/admin/card-matching
```

### 8.2 照合の流れ（毎月月初に実施）

```
1. MFビジネスカード管理画面 → 前月分の利用明細CSVをダウンロード
2. 照合UI（/admin/card-matching）を開く
3. 月を選択（前月）→ CSVをドロップまたはファイル選択
4. 自動照合が実行される（5-10秒）
5. 4つのタブを順に確認・処理
```

### 8.3 照合結果の4タブ

| タブ | 内容 | 対応 |
|------|------|------|
| **自動照合済み** | 購買申請と明細が自動マッチした件 | 確認のみ（差額ありは調整仕訳がドラフト作成済み。MF会計Plusで確認・承認） |
| **要確認** | 候補が複数あり判断が必要な件 | 正しい明細を選んで [これに確定]（差額がある場合は調整仕訳を自動作成） |
| **明細なし** | 申請はあるが明細にない件 | [来月もう一度照合] or [キャンセル済み] |
| **未申請利用** | 明細はあるが申請がない件 | [本人に確認] or [経費で処理] |

### 8.4 マッチングの仕組み

**Phase 1: 予測マッチング（高精度）**
- 購買申請の承認時に、カード下4桁・金額・予想利用日を「予測テーブル」に自動記録
- 実際のカード明細と予測テーブルを照合（カード番号×金額×日付7日以内）
- 金額完全一致 → 自動確定、5%以内 → 自動確定（差額調整付き）、5-10% → 要確認

**Phase 2: スコアリング（フォールバック）**
- 予測でマッチしなかった明細をMF会計Plusの仕訳と照合
- 金額(50点) + 日付(30点) + 加盟店名(20点) = 100点満点
- 80点以上 → 自動確定、50-79点 → 要確認、50点未満 → 未マッチ

### 8.5 引落照合

```
1. 照合UIの「引落照合」タブを開く
2. MFビジネスカード管理画面 → 入出金履歴CSVをダウンロード
3. CSVをドロップまたは貼り付け → [照合を実行]
4. 未払金(請求)合計 と 引落額 を比較
5. 差額がある場合 → 原因ガイドに従って調査
```

**よくある差額原因:**
- 月末利用分が翌月確定に繰越 → 翌月の引落に含まれる
- 返品・キャンセルで引落額が減額 → 対応する貸方仕訳を確認
- 年会費など未払金計上外の引落 → 引落明細の摘要を確認

### 8.6 予測テーブルの仕組み

| タイミング | 処理 |
|-----------|------|
| 購買承認時 | カード払い → 従業員のカード下4桁を解決 → GAS「予測カード明細」シートに予測レコード作成 |
| 出張申請時 | 交通費(trip_transport)と宿泊費(trip_hotel)を別行で予測レコード作成（日当は非カードのため対象外） |
| 月次照合時 | 予測テーブル + MF仕訳 + CSVカード明細 → マッチングエンジンで照合 |
| 照合確定時 | 予測レコードのステータスを `matched` に更新 |

**前提**: 従業員マスタのG列(card_last4)・H列(card_holder_name)にカード情報が登録されていること

## 10. ステータス遷移図

### 通常フロー（カード・請求書共通）

```
申請済 → 承認済 → 発注済 → 検収済 → 証憑完了 → 計上済 → 支払済
```

### 部分検収フロー（分割納品）

```
発注済 → 部分検収(3/10) → 部分検収(7/10) → 検収済(10/10) → 証憑完了 → 計上済
```

### 購入済フロー（立替）

```
申請済 → 承認済 → 検収済（即時） → 証憑完了 → 計上済 → 支払済
```

### 差戻し・取消・返品

```
申請済 → 差戻し（DMに再申請リンク付き — 新PO番号で再申請）
申請済 → 取消（終了）
承認済 → 取消（発注前に限る）
検収済 → 返品（全量。取消仕訳を自動ドラフト作成。MF会計Plusで確認・承認）
検収済 → 一部返品（数量指定。按分金額で取消仕訳を自動ドラフト作成）
```
