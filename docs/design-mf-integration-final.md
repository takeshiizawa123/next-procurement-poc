## 1. 基本方針

### 1.1 経路の使い分け

```
支払方法で自動分岐:

  会社カード → 案G: Google Drive証憑 + MF会計Plus API仕訳
  請求書払い → 案G: Google Drive証憑 + MF会計Plus API仕訳
  従業員立替 → 案C: MF経費で立替精算
```

### 1.2 カード利用の全量管理

```
従業員カード（1人1枚）:
  全利用を購買システムで管理
  ├── 購買 → /purchase → PO番号
  ├── 出張 → /trip    → TR番号
  └── 上記以外 → ★未申請アラート（統制強化）

管理本部カードA（購買用）:
  └── 購買 → /purchase → PO番号

管理本部カードB（サブスク用）:
  └── 固定仕訳ルール（MF会計Plusの自動仕訳で処理）
```

### 1.3 MF経費の役割

```
MF経費の用途:
  ○ 従業員立替の精算（振込データ生成）← 唯一の用途
  × 購買の証憑管理
  × 出張旅費の申請
  × カード利用の管理
```

## 4. カード明細マッチング

### 4.1 方式B: 予測テーブルマッチング

#### 基本思想

```
従来方式（スコアリングのみ）:
  カード明細 × 購買台帳 → 金額・日付・取引先名でスコアリング
  問題: 同日同額の複数購入、取引先名の表記揺れで精度が下がる

方式B（予測テーブル）:
  購買申請時に「予測カード明細」を生成
  → カード番号(下4桁) × 金額 × 利用予定日 で事前に照合キーを確定
  → 実際のカード明細到着時に、予測と突合するだけ
  → 「誰のカードで」「いくら」「いつ」が事前に分かるため高精度
```

#### なぜ方式Bか

```
方式A（スコアリングのみ）の限界:
  ① 同日同額同店の複数購入 → 区別不能
  ② 加盟店名の表記揺れ → エイリアス辞書の継続メンテが必要
  ③ 出張の複数明細（交通+宿泊+日当）→ 個別マッチング困難

方式B（予測テーブル）の優位性:
  ① カード番号で「誰の利用か」が確定 → 同日同額でも区別可能
  ② 金額+日付の完全一致で照合 → 取引先名に依存しない
  ③ 出張申請時に予測明細を分割生成 → 個別マッチング可能
  ④ スコアリングはフォールバックとして残す → 予測外の利用も拾える
```

#### マッチングの全体フロー

```
[申請時]
  /purchase or /trip 申請
    → 申請者のcard_last4を従業員マスタから取得
    → predicted_card_transactions に予測明細を登録
       (card_last4, predicted_amount, predicted_date, po_number)

[月次照合時]
  ① GET /journals?entered_by=none → Stage 2仕訳を取得
  ② 各Stage 2仕訳の摘要からカード番号下4桁を抽出
     （MF会計Plus自動仕訳ルールの摘要: "HIROSHI OKA *3815 AMAZON.CO.JP"）
  ③ card_last4で予測テーブルを絞り込み
  ④ 絞り込んだ中で金額×日付の完全一致 → 確信マッチ
  ⑤ 完全一致なし → スコアリング（従来方式）にフォールバック
  ⑥ それでも未マッチ → 未申請アラート
```

```

入力:
  tr_number: "TR-2026-0015"
  applicant: "田中太郎"
  departure: "2026-04-10"
  return: "2026-04-12"
  destination: "大阪"
  items:
    - type: "transport", description: "新幹線 東京-新大阪 往復", amount: 27500
    - type: "hotel", description: "大阪ホテル 2泊", amount: 24000

  ① 各明細項目ごとに予測明細を生成
  ② 交通費の利用予定日を推定:
     - 往路: departure（4/10）

  ③ 宿泊の利用予定日を推定:
     - チェックイン日（departure）に全額決済が一般的

    po_number: "TR-2026-0015",

    card_last4: "3815",
    predicted_amount: 27500,
    predicted_date: "2026-04-10",   // 往路日
    supplier: "JR東海",
    status: "pending"
  },
  {
    id: "PCT-2026-0091",
    po_number: "TR-2026-0015",
    type: "trip_hotel",
    card_last4: "3815",
    predicted_amount: 24000,
    predicted_date: "2026-04-10",   // チェックイン日
    supplier: "大阪ホテル",
    status: "pending"

  - 交通費が往復別決済の場合は、従業員が /trip で明細を分けて登録
  - 宿泊が複数ホテルの場合も同様に分けて登録
  - 金額が確定しない場合（概算）は predicted_amount に概算額を入れ、
    差額はスコアリングで吸収（5%以内は確信マッチ）
```

### 4.4 マッチングアルゴリズム（予測テーブル統合版）

#### アーキテクチャ

```
MF会計Plus:
  自動仕訳ルール:
    従業員カード・管理本部カードAの全明細
      → 未払金(MFカード:未請求) / 未払金(MFカード:請求)
      → entered_by: none として仕訳登録
      → 摘要に "カード名義 *下4桁 加盟店名" が自動設定

    管理本部カードBの明細
      → 費用科目 / 未払金(MFカード:請求)（加盟店名で科目判定）

購買システム（月次バッチ）:
  ① GET /journals?entered_by=none → Stage 2仕訳を全件取得
  ② 各仕訳の摘要からカード番号下4桁を抽出（parseCardLast4）
  ③ predicted_card_transactions から status=pending を取得
  ④ Phase 1: 予測マッチング（card_last4 × 金額 × 日付）
  ⑤ Phase 2: スコアリング（Phase 1で未マッチのもの）
  ⑥ 結果処理
```

#### Phase 1: 予測マッチング（高精度・高速）

```typescript
// 摘要からカード下4桁を抽出
function parseCardLast4(remark: string): string | null {
  // "HIROSHI OKA *3815 AMAZON.CO.JP" → "3815"
  const match = remark.match(/\*(\d{4})\s/);
  return match ? match[1] : null;
}

// Phase 1: 予測テーブルとの突合
function predictiveMatch(
  stage2Journals: Stage2Journal[],
  predictions: PredictedTransaction[]
): MatchResult[] {
  const results: MatchResult[] = [];

  for (const journal of stage2Journals) {
    const cardLast4 = parseCardLast4(journal.remark);
    if (!cardLast4) continue;

    // 同じカード番号の予測明細に絞り込み
    const candidates = predictions.filter(p =>
      p.card_last4 === cardLast4 && p.status === "pending"
    );

    for (const pred of candidates) {
      const amountDiff = Math.abs(journal.amount - pred.predicted_amount);
      const amountDiffRate = amountDiff / pred.predicted_amount;
      const dateDiff = Math.abs(daysBetween(journal.date, pred.predicted_date));

      // 金額完全一致 + 日付7日以内 → 確信マッチ
      if (amountDiff === 0 && dateDiff <= 7) {
        results.push({
          type: "confident",
          prediction: pred,
          journal,
          score: 100,
          amountDiff: 0
        });
        pred.status = "matched";  // 使用済みにマーク
        break;
      }

      // 金額5%以内 + 日付3日以内 → 確信マッチ（差額あり）
      if (amountDiffRate <= 0.05 && dateDiff <= 3) {
        results.push({
          type: "confident_with_diff",
          prediction: pred,
          journal,
          score: 90,
          amountDiff: journal.amount - pred.predicted_amount
        });
        pred.status = "matched";
        break;
      }

      // 金額10%以内 + 日付7日以内 → 候補マッチ
      if (amountDiffRate <= 0.10 && dateDiff <= 7) {
        results.push({
          type: "candidate",
          prediction: pred,
          journal,
          score: 70,
          amountDiff: journal.amount - pred.predicted_amount
        });
        // candidateはbreakしない（複数候補あり得る）
      }
    }
  }
  return results;
}
```

#### Phase 2: スコアリング（フォールバック）

```
Phase 1で未マッチのStage 2仕訳に対して、従来のスコアリングを実行

totalScore = amountScore(50) + dateScore(30) + nameScore(20)

金額スコア（最大50点）:
  差額率 = |購買金額 - カード金額| / 購買金額
  一致(0%):     50点
  1%以内:       40点（税端数・為替微差）
  5%以内:       25点（送料追加等）
  10%以内:      10点（要確認）
  10%超:         0点

日付スコア（最大30点）:
  同日: 30 / 1日差: 25 / 2日差: 20 / 3日差: 15 / 4-7日: 5 / 8日超: 0

取引先名スコア（最大20点）:
  正規化後一致: 18 / 部分一致: 15 / 先頭3文字一致: 8 / 不一致: 0

判定:
  ≥80: 確信マッチ（自動処理）
  50-79: 候補マッチ（経理確認）
  <50: 未マッチ

適用ケース:
  - 予測テーブルに登録がない利用（申請前のカード利用等）
  - card_last4が摘要から抽出できなかった場合
  - 管理本部カードA（購買用・個人紐付けなし）の利用
```

function normalizeMerchantName(name: string): string {

    .toUpperCase()

### 4.7 マッチング結果の処理

#### 確信マッチ（金額一致）

```
処理:
  → predicted_card_transactions 更新:
    status: "matched"
    matched_journal_id: 99001
    matched_at: "2026-04-01T10:00:00"
    amount_diff: 0
  → GAS更新: stage=2, stage2_journal_id=99001, card_matched=true
  → 追加仕訳は不要（Stage 2仕訳は自動仕訳ルールで登録済み）
```

#### 確信マッチ（差額あり）

```
例: 予測 ¥33,000 / カード実績 ¥33,550（差額+¥550）

処理:
  → predicted_card_transactions 更新:
    status: "matched"
    matched_journal_id: 99002
    amount_diff: +550
  → GAS更新: stage=2, stage2_journal_id=99002
  → 差額調整仕訳を自動作成:

  POST /journals:
    tags: ["PO-2026-0043", "Stage2-adj"]
    memo: "PO-2026-0043 差額調整 +¥550"
    branches:
      - debitor: 消耗品費 ¥550（Stage 1と同じ費用科目）
        creditor: 未払金(MFカード:未請求) ¥550

  → Stage 1の未請求残高が¥33,550に増え、Stage 2の振替額と一致
```

#### 未マッチのStage 2仕訳

```
従業員カード・管理本部カードAの明細で、予測テーブルにもスコアリングにもマッチしない
  → ★未申請アラート
  → Slack通知: "⚠️ 未申請のカード利用を検出: 3/18 ¥1,200 TAXI JAPAN (*3815 田中太郎)"
  → 経理が確認:
    - 申請漏れ → 従業員に申請を依頼（カード番号から特定可能）
    - 不正利用 → 対応
    - 正当な利用（購買システム外） → MF会計Plusで仕訳を修正
      （未払金:未請求/未払金:請求 → 費用科目/未払金:請求）

予測テーブルとの違い（従来方式比）:
  → card_last4 から「誰のカードか」が即座に判明
  → 従業員への確認が迅速化
```

#### 未マッチの予測明細

```
predicted_card_transactions に status=pending のまま残ったもの
  → カード確定値がまだ出ていない（翌月繰越）
  → or キャンセル・返品
  → or 現金払いに変更（カード未使用）
  → 翌月に再照合。2ヶ月連続未マッチは status="unmatched" + アラート

対処:
  ① 翌月繰越: そのまま pending で翌月照合に含める
  ② キャンセル: status="cancelled" に手動更新
  ③ 支払方法変更: status="cancelled" → 正しい支払方法で再申請
```

### 4.8 マッチング精度の期待値

```
予測マッチ（Phase 1）で解決:
  ├── 従業員カード × 購買申請済み → 90%以上が金額完全一致で確信マッチ
  ├── 従業員カード × 出張申請済み → 70-80%（宿泊の金額変動あり）
  └── 管理本部カードA → 予測テーブルなし（Phase 2へ）

スコアリング（Phase 2）で解決:
  ├── 管理本部カードAの利用 → 従来通りのスコアリング
  ├── 予測テーブルにない利用 → 加盟店名で照合
  └── card_last4抽出失敗 → 全件スコアリング

最終的な未マッチ率: 月間5%以下を目標
  → 未マッチ = 未申請アラートとして統制に活用
```

## 13.5 消費税 仕入税額控除の区分方針

### 前提（第12期 消費税申告書に基づく）

| 項目 | 値 |
|------|-----|
| 課税方式 | 原則課税 |
| 課税標準額 | 338,723,000円（約3.39億円） |
| 課税売上割合 | 99.9% |
| 非課税売上 | 受取利息 105,059円のみ |
| 5億円超 | 翌期以降に可能性あり |

### 現行方針（課税売上高5億円以下の間）

- 全額控除が適用されるため、**課税/共通/非課税の区分は不要**
- システムでは全件「**課税仕入10%**」（or 軽減8%）で仕訳を作成
- 税率の判定は証憑のOCR読取結果に基づく（10% or 8%軽減）
- 区分の自動判定ロジックは**実装しない**

### 課税売上高5億円超になった場合

- 個別対応方式 or 一括比例配分方式 → **顧問税理士と相談して決定**
- **一括比例配分方式の場合**: システム変更不要（区分不要、全額×課税売上割合で控除）
- **個別対応方式の場合**: 決算時にMF会計Plusで部門ベースの税区分一括振替で対応
  - 営業・開発・製造・ロジスティクス → 課税仕入（課税売上対応）
  - 管理本部・取締役・共通 → 共通課税仕入
  - 購買管理システム側の変更は不要（MF会計Plus側で対応）
- 事前届出: いずれの方式も**届出不要**（申告時に選択）
- 一括比例は**2年間の継続適用義務**あり

### 購買管理システムの対応

- **現行のまま変更不要**
- 仕訳に部門コードを確実に記録する（将来の一括振替に備えて）
- OCRで税率（10%/8%）を読み取り、仕訳に反映する

### H3問題: MFビジネスカード→MF経費の連携（調査結果 2026-03-27）

```
問題:
  ハイブリッド方式ではMF経費を「従業員立替のみ」に限定した。
  しかしMFビジネスカードの利用明細がMF経費に自動連携されると、
  従業員が購買分のカード明細を誤ってMF経費で経費申請する
  「重複申請リスク」があるのでは？

調査結果:
  ① MFビジネスカード→MF経費の連携は「手動選択方式」
     → 明細は自動で経費申請に含まれない
     → 従業員が「連携サービスから登録」画面で明細を手動選択して初めて登録される
     → したがって、自動的な重複申請は発生しない

  ② 管理者がカード単位・従業員単位で連携を制御可能
     → 管理設定→経費機能設定→コーポレートカードで設定
     → 特定カードの連携を解除することも技術的に可能

  ③ 連携自体の停止（解除）も可能
     → ただし再連携が必要になる場合の手間あり
     → 「連携解除」ボタンの使用は非推奨とMF公式に記載

結論: 重複申請リスクは当初の懸念より低い
  → 明細は手動選択方式のため、自動的な重複は起きない
  → 以下の運用ルール周知で対応可能（連携停止は不要）

運用ルール（従業員向け周知事項）:
  ① MF経費の「連携サービスから登録」でカード明細を選択しないこと
  ② MF経費で登録するのは「従業員立替」の経費のみ
  ③ カード決済の購買・出張は購買システム（/purchase, /trip）で申請済み
  ④ 判断に迷う場合は管理本部に確認

追加対策（Phase 2で検討）:
  → MF経費の管理設定で、購買用カードの連携を解除する
  → これによりMF経費に購買用カードの明細が表示されなくなる
  → 立替精算にカードは使わないため、連携解除による影響なし

参考URL:
  - MFビジネスカード連携ガイド: biz.moneyforward.com/support/expense/guide/linkage-general/li13.html
  - コーポレートカード連携手順: biz.moneyforward.com/support/expense/guide/admin-ex/ex42.html
```

| **税抜金額** | 仕訳の本体金額 | **新規** |
| **登録番号（T+13桁）** | 適格請求書の検証 | **新規** |

### 適格請求書の検証フロー

```
証憑添付 → Gemini OCRで登録番号を読取
  → 登録番号あり → 国税庁Web-API（invoice-kohyo.nta.go.jp）で検証
    → 有効 → 「適格請求書: T1234567890123（○○株式会社）」をスレッドに表示
    → 無効/未登録 → ⚠️ OPSチャネルに警告通知
  → 登録番号なし & 請求書 → ⚠️「登録番号なし」OPSチャネルに警告通知
  → 登録番号なし & 納品書/領収書 → 警告なし（登録番号記載は任意）
```
