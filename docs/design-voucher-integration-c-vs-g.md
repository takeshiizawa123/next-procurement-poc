**作成日**: 2026-03-27
**目的**: 具体的な運用シナリオで案C（MF経費経由）と案G（Google Drive）の日常・月次・例外処理を比較し、最適な方式を選定する

## 0. 共通前提

### 登場人物
- **田中さん**（一般社員）: Amazonでモニターを¥52,800で購入
- **管理本部 鈴木さん**（経理担当）: 月次で仕訳確認・承認を実施

### 購買フローの共通部分（案C/G共通）

```
田中さん                    購買システム（Slack Bot）
────────                  ─────────────────────
① /purchase 52800
   モニター Amazon           → GASに申請登録（PO-2026-0042）
                             → Slack承認リクエスト送信
② 鈴木さんが [承認] 押下     → GAS更新: 承認済
③ 田中さんがAmazonで決済     → GAS更新: 発注完了
④ 商品到着 → [検収完了]      → GAS更新: 検収済
⑤ 田中さんがSlackスレッドに
   納品書PDFを添付            → Gemini OCRで金額照合
                             → GAS更新: 証憑=添付済
                             ────────────────────
                             ↓ ここから案C/Gで分岐 ↓
```

### API制約の要約

```
MF会計Plus API:
  POST /journals           → 仕訳作成 ✅（証憑フィールドなし）
  GET /vouchers/{id}/file  → 証憑ダウンロード ✅（読取のみ）
  POST /vouchers           → ❌ 存在しない

MF経費 API:
  POST upload_receipt      → 証憑UP + OCR明細自動生成 ✅
  POST ex_transactions     → 経費明細作成 ✅
  POST ex_reports          → ❌ 存在しない（申請はUI手動）
```

```

## 3. カード明細の対象外処理（両案共通の重要タスク）

### 3.1 問題の構造

```
MFビジネスカードでの決済
  │
  ├──→ MF会計Plus「通帳・カード他」に明細が自動で出現
  │    → これを放置すると、カード明細からも仕訳候補が生成される
  │    → 購買システム経由の仕訳と二重になる
  │
  └──→ MF経費にも明細が自動連携
       → 案C: 購買分の明細が既にupload_receipt経由で存在するため二重
       → 案G: MF経費を経由しないので、カード明細は放置でOK
```

### 3.2 対象外処理の手順

```
鈴木さんの作業（月次・両案共通）
─────────────────────────────

① 購買システムの月次照合レポートを取得
   | PO番号       | 取引日     | 金額    | 取引先     | カード利用日  |
   |-------------|-----------|---------|-----------|------------|
   | PO-2026-0042 | 2026-03-15 | ¥52,800 | Amazon    | 2026-03-15  |
   | PO-2026-0043 | 2026-03-18 | ¥33,000 | モノタロウ  | 2026-03-19  |

② MF会計Plusの「通帳・カード他」画面を開く

③ 照合レポートの金額・日付を元にカード明細を特定
   → 金額一致 + 日付±3日 + 加盟店名で照合

④ 該当する明細を「対象外」に設定
   → MF会計Plusの自動仕訳ルールで「加盟店名にAmazonを含む → 対象外」
      のようなルールを事前設定しておくと効率化できる

⑤ 未照合の明細を確認
   → 購買台帳にないカード利用 = 一般経費 or 不正利用
   → 一般経費分は従来通りの処理（MF経費経由 or 仕訳候補から登録）
```

### 3.3 案C vs 案G: カード明細処理の差

| 場所 | 案C | 案G |
|------|-----|-----|
| MF会計Plus カード明細 | 対象外処理（必須） | 対象外処理（必須） |
| MF経費 カード明細 | 購買分と一般経費の仕分けが必要 | **放置でOK**（MF経費を経由しない） |

**案Gの方がシンプル**: MF経費内でのカード明細仕分けが不要。

## 8. 支払方法別フロー分析

### 8.1 支払方法3パターン

購買台帳の `payment_method` は3種類: **会社カード / 請求書払い / 立替**

| 支払方法 | 仕訳の貸方 | カード明細 | MF経費に明細連携 | 精算方法 |
|---------|-----------|-----------|---------------|---------|
| 会社カード | 未払金(MFカード) | ✅ 出る | ✅ 自動連携 | カード引落（自動） |
| 請求書払い | 買掛金 | ❌ 出ない | ❌ なし | 振込（経理が実行） |
| **従業員立替** | **未払金(従業員)** | **❌ 出ない** | **❌ なし** | **従業員への振込精算** |

### 8.2 パターン別: 会社カード決済（月間の大半）

```
フロー: /purchase → 承認 → カード決済 → 検収 → 証憑添付

仕訳:
  借方: 消耗品費    ¥52,800
  貸方: 未払金(MFカード:未請求)  ¥52,800

後続:
  → カード利用明細到着 → Stage 2 照合
  → 銀行引落 → Stage 3 消込
```

| | 案C | 案G |
|--|-----|-----|
| 証憑処理 | upload_receipt → MF経費 | Drive保存 → 仕訳ドラフト |
| カード明細処理 | MF経費+MF会計Plusで対象外 | MF会計Plusで対象外 |
| 特記事項 | なし | なし |

### 8.3 パターン別: 請求書払い

```
フロー: /purchase → 承認 → 発注 → 検収 → 請求書受領 → 証憑添付

仕訳:
  借方: 消耗品費    ¥330,000
  貸方: 買掛金      ¥330,000

後続:
  → 支払期日に振込 → 買掛金消込

### 8.4 パターン別: 従業員立替（★ 重要な差が出る）

```
フロー: /purchase → 承認 → 田中さんが自腹で購入 → 検収 → 証憑添付

仕訳:
  借方: 消耗品費          ¥8,800
  貸方: 未払金(従業員立替)  ¥8,800

後続:
  → 田中さんへの立替精算（振込）
  → 未払金(従業員立替) / 普通預金 ¥8,800
```

#### 案Cの場合（従業員立替）

```

  → 田中さんの立替経費として明細が存在

  → MF経費の「精算」機能で田中さんへの振込データを生成可能
  → 振込実行 → 精算完了

メリット:
  ✅ MF経費の精算機能がそのまま使える
  ✅ 仕訳 + 精算振込がMF経費内で一元管理
  ✅ 田中さんへの振込履歴がMF経費に残る
```

#### 案Gの場合（従業員立替）

```
購買システム → Drive保存 → POST /journals（仕訳ドラフト）

★ 仕訳は作れるが、精算管理が宙に浮く:
  → 仕訳: 消耗品費 / 未払金(従業員立替)  ✅
  → 田中さんへの振込精算は？
     選択肢1: MF経費を別途使う
       → 田中さんがMF経費で立替申請を別途提出 → 二度手間
       → or 経理がMF経費で代行作成 → 案Cと同じ（MF経費を使う）
     選択肢2: 購買システムで精算管理を実装
       → スコープが大幅に拡大（精算台帳・振込データ生成）
     選択肢3: 手動で振込
       → 管理が散らばる。証跡が購買台帳にしか残らない

どの選択肢でもMF経費を使うことになる or スコープ拡大
```

### 8.8 MFカード決済の混在（案C固有の問題）

案Cの場合、MFビジネスカードの利用明細がMF経費に自動連携されるため:

```
MF経費の明細一覧:
  ├── [カード自動連携] 03/15 Amazon ¥52,800     ← MFカードから自動
  ├── [API作成]       03/15 PO-2026-0042 ¥52,800 ← 購買システムから
  └── 同じ取引に対して2件の明細が存在する

経理の仕分け作業:
  → カード自動連携の明細は放置（申請に含めない）
  → API作成の明細のみ申請に含める
  → PO番号の有無で識別可能だが、件数が多いと面倒
```

案Gではこの問題が**発生しない**（MF経費を経由しないため）。
