# 統合経費管理 業務フロー設計（B案）

**版数**: v2.0
**最終更新**: 2026-04-12（Supabase移行・予約完了申請・仕訳Stage設計を反映）
**方針**: 購買管理システム主導でMF経費→MF会計Plus連携を制御

---

## 1. システムの主要コンポーネント

| コンポーネント | 役割 |
|---------------|------|
| **購買管理システム** | 申請・承認・照合・仕訳作成のすべての司令塔（自社開発・Vercel + Supabase Postgres） |
| **MFビジネスカード** | 従業員1人1枚のバーチャルカード。すべての経費決済に使用 |
| **MF経費** | カード明細の取り込み口。購買管理がAPIで明細を取得（ユーザーは触らない） |
| **MF会計Plus** | 仕訳の最終出力先。購買管理がAPIで仕訳を直接登録 |
| **Slack** | 通知・承認の窓口（DMボタンで承認、申請完了通知等） |
| **Webアプリ** | 申請・マイページ・管理画面のメインUI |

## 2. 全体構成 — データの流れ

```
Layer 1: ユーザー操作
┌──────────────────────────┐  ┌─────────────────────┐
│ Webアプリ                 │  │ Slack                │
│ 申請・マイページ・管理画面 │  │ 承認DMボタン・通知    │
└───────────┬──────────────┘  └──────────┬──────────┘
            ↓                            ↓
Layer 2: 購買管理システム（司令塔）
┌──────────────────────────────────────────────────┐
│ Next.js (Vercel) + Supabase Postgres (Tokyo)      │
│ Drizzle ORM / 15テーブル / PgBouncer接続           │
│                                                    │
│ ・申請・承認                                       │
│ ・予測テーブル（predicted_transactions）            │
│ ・照合エンジン（card-matcher-v2: office_member_id） │
│ ・仕訳作成（MF会計Plus API直接登録）               │
│ ・出張統制（差異検知/未申請/重複/部門コスト）       │
└───────────────────────┬──────────────────────────┘
                        ↓
Layer 3: 外部システム（バックエンド連携のみ）
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ MFビジネス    │  │ MF経費        │  │ MF会計Plus   │
│ カード        │→│ カード明細     │→│ 仕訳の       │
│ 各従業員1枚   │  │ 自動取込      │  │ 最終出力先   │
│ 全決済に使用  │  │ API取得       │  │ API登録      │
└──────────────┘  └──────────────┘  └──────────────┘
```

決済 → MF経費（自動取込）→ 購買管理がAPI取得 → 照合 → MF会計Plus に仕訳登録

---

## 3. 経費の3パターン — すべて統一フロー

| パターン | 内容 | 決済方法 |
|---------|------|---------|
| **購買** | 事前申請して物品・サービスを購入 | MFバーチャルカード |
| **出張** | 予約完了後に申請（事後承認） | MFバーチャルカード |
| **立替** | やむを得ず個人で立替 → 精算 | 現金/個人カード → 給与精算 |

いずれも Webアプリ/Slackで申請 → 部門長承認 → カード決済 → 自動照合 → 仕訳

---

## 4. 申請フロー

```
① 申請（Webアプリ or Slack）
  ↓
② 予測テーブル登録（predicted_transactions に保存）
  ・従業員ID + 金額 + 予想日付 + タイプ(purchase/trip_transport/trip_hotel/reimbursement)
  ↓
③ 部門長承認（Slack DMで [承認] [差戻し] ボタン）
  ↓
④ 申請者通知（承認完了をSlackで通知）
  ↓
⑤ 予測テーブル状態更新（「approved」に変更、カード決済を待つ状態）
```

---

## 5. カード決済 → 明細取得

```
従業員: バーチャルカードで決済（Amazon/スマートEX/じゃらん等）
  ↓
MFビジネスカード: 決済データ自動連携
  ↓
MF経費: カード明細として取り込み、office_member_id で従業員を自動紐付け
  ↓
購買管理システム: fetchCardStatements API で取得（週次cron + 任意手動実行）
```

---

## 6. 照合エンジン — 予測テーブル × カード明細

### 照合キー

`office_member_id`（従業員）× 金額 × 日付（±7日）

> GAS時代は card_last4 ベースだったが、Supabase移行後は office_member_id ベースに変更（card-matcher-v2）。
> これにより、カード番号変更時の再設定が不要になり、照合精度が向上。

### 照合例

**予測テーブル（申請データ）:**

| PO番号 | 従業員 | 金額 | 予想日 | タイプ |
|--------|--------|------|--------|--------|
| PO-202604-0001 | 田中 | ¥14,000 | 2026-04-15 | trip_transport |
| PO-202604-0002 | 田中 | ¥30,000 | 2026-04-15 | trip_hotel |
| PO-202604-0003 | 鈴木 | ¥3,500 | 2026-04-15 | purchase |

**MF経費カード明細:**

| 加盟店 | 金額 | 日付 | 従業員 |
|--------|------|------|--------|
| EX東海 | ¥14,000 | 2026-04-15 | 田中 |
| じゃらんnet | ¥29,800 | 2026-04-15 | 田中 |
| Amazon.co.jp | ¥3,500 | 2026-04-15 | 鈴木 |

### 照合結果

| 結果 | 処理 |
|------|------|
| 完全一致 | → 自動仕訳（Stage 2） |
| 差額あり（±5%以内） | → 自動（差額調整仕訳付き） |
| 差額あり（5-10%） | → 要確認（管理者手動） |
| 申請なし | → 未申請利用アラート |
| 明細なし | → 督促（翌月再照合） |

## 7. 仕訳の作成 — 3段Stage

MF会計Plusの自動仕訳ルールは使わず、購買管理が直接APIで仕訳を作成（B案の核心）。

### Stage 1: 費用認識（申請承認時）

```
借: 旅費交通費    14,000
貸: 未払金:MFカード:未請求  14,000
```
→ 出張申請承認時に作成（経費認識）

### Stage 2: 請求確定（カード明細マッチング後）[未自動化]

```
借: 未払金:MFカード:未請求  14,000
貸: 未払金:MFカード:請求    14,000
```
→ カード明細とマッチング後に作成（請求確定）

### Stage 3: 支払完了（銀行引落時）[未自動化]

```
借: 未払金:MFカード:請求  14,000
貸: 普通預金              14,000
```
→ 銀行引落時に作成（支払完了）

**摘要の自動生成:**
`東京→大阪 新幹線のぞみ ¥14,000 申請者:田中 (PO-202604-0001)`

> **現在の実装状況**: Stage 1は仕訳ドラフトとして実装済み。Stage 2・3は今後自動化予定。

## 8. B案の特徴

### メリット

- 自動仕訳ルールの設定が不要 — 経理担当者の負担ゼロ
- 新規加盟店が出ても自動対応 — 運用保守がほぼ不要
- 仕訳の摘要・補助科目を自由にカスタマイズ可能
- 二重仕訳のリスクなし — 仕訳作成は購買管理だけ
- MF経費 → 購買管理 → MF会計Plus の経路が一本化
- office_member_id で従業員を確実に特定（card_last4不要）

### 事前設定

- MFビジネスカード × MF経費 の自動連携設定（各従業員のバーチャルカードをMF経費アカウントに紐付け）
- MF会計Plusの自動仕訳ルールは「使わない（OFF設定）」（二重仕訳防止）
- MF経費APIアクセストークンの発行
- MF会計Plus OAuth認証
- 従業員マスタとMF経費 office_member の紐付け確認

## 9. 例外フロー

| パターン | 発生例 | 対応方針 |
|---------|--------|---------|
| 申請なしの利用 | 出張中に急遽タクシー利用 | 管理者にアラート → 事後申請を促す |
| 概算と実額の差 | 新幹線予約変更で金額変更 | 差額±5%以内は自動承認、超える場合は管理者確認 |
| 申請したのに明細なし | 予約後にキャンセル | 1週間後に申請者に確認DM |
| 重複決済 | 同日同額の購買と出張 | office_member_id + 申請内容で個別マッチング |
| 返金・キャンセル | 出張キャンセルでカード返金 | 負の値の明細として取り込み、Stage 1の取消仕訳を作成 [未自動化] |
| 不明な明細 | 従業員特定不可 | 管理者ダッシュボードに「未照合」として表示、手動対応 |

## 10. 役割分担

| 役割 | やること | やらないこと |
|------|---------|------------|
| 申請者（従業員） | Webアプリ/Slackで申請 → カード決済 → 検収 | MF経費・MF会計Plusに触る |
| 承認者（部門長） | Slack DMで承認/差戻し | 個別の照合・仕訳確認 |
| 管理本部 | 未照合明細の確認、例外対応、月次締め | 通常の仕訳作成・自動仕訳ルール管理 |
| 購買管理システム | 申請受付・予測登録・照合・仕訳自動生成 | ― |
| MFビジネスカード | 決済・明細をMF経費に連携 | ― |
| MF経費 | カード明細の保管・API提供 | 申請承認・仕訳作成 |
| MF会計Plus | 仕訳の最終保管・経理出力 | 自動仕訳ルール（OFF） |
